# 网课视频逐字稿 -> 高质量 Markdown 笔记 Agent 系统

## 项目目标

将网课视频的逐字稿（可达 6w 字），结合课程代码和课件资料（Word/Markdown），
自动生成高质量、不遗漏知识点的 Markdown 笔记。

## 核心设计哲学

- 不要把结构理解这个最难的认知任务分散到小块中让 LLM 盲人摸象。
  推迟结构理解到拥有完整干净文本之后，一次性交给大窗口模型。
- 每个 LLM 调用只干一件事：纠错就是纠错，润色就是润色，结构化就是结构化。
  分离关注点，质量自然上去。
- 每块 LLM 必须拥有上下文视野。通过全局摘要 + 前文首尾 + 后文开头 + 位置标签的组合，
  四层上下文各解决一个问题，全部零额外 LLM 成本。
- 代码必须嵌入叙事流。课程笔记的核心价值是老师先讲思路->再写代码->再解释的渐进过程，
  不是附录式的代码堆砌。

## 阶段0：预处理

### 0.0 纯正则噪音清洗（纯程序，一次性，零LLM）

**目的**：在纠错和润色之前，用零成本的正则规则去除确定无疑的噪音，减少后续所有步骤的 token 消耗。

**清洗规则**：

| 去掉的内容 | 方法 |
|---|---|
| 纯填充词 | 正则匹配“那个”、“就是说”、“然后呢”、“对吧”、“是不是” |
| 重复口头禅 | 连续出现 3 次以上的短模式（如“嗯嗯嗯”） |
| 多余空行 | `\n{3,}` → `\n\n` |
| 独立语气叹词 | 行首/行尾的“啊”、“哦”、“呃”（不在词语中间） |

**原则**：保守清洗——只去掉确信无疑的噪音，边缘情况留给阶段1 LLM 处理。预估可节省 5%-10% 的 token。

### 0.1 术语词典抽取（可选，纯程序，一次性，零LLM）

**目的**：从代码/课件中提取技术术语，作为阶段0.2纠错的参考词表。

**方案**：扫描代码文件和课件，程序化提取：
- 类名、接口名、枚举名、方法名、函数名
- 注解（@Autowired、@RestController）
- import/require 中的包名
- 配置文件中的 bean 名、属性名
- 课件中的技术关键词

去噪合并去重，生成约 300-500 词的术语表。代码库变化时重新扫一遍即可。

**无课件/代码时**：跳过此步骤。纠错阶段依赖 LLM 自身技术术语知识。

### 0.2 全文错别字纠正（便宜LLM）

**目的**：在润色之前先把语音识别的错别字修好，保证后续 RAG 检索的准确性。

**方案**：
1. 将原始逐字稿按 ~10000 字切成大块
2. 每块输入 = [术语词典（如果有）] + [高频疑似术语词频统计] + [原始大块]
3. 使用 DeepSeek-V4-Flash（non-thinking，1M窗口，$0.14/M入 $0.28/M出）
4. 任务单一：修正技术术语的错别字，不改结构、不去口语、不加格式
5. Prompt 约束："不要改动非技术术语的文字，不要把中文改成英文，只修正错误的技术术语音译"

**纠错 vs 润色的本质区别**：纠错是机械性替换任务，对注意力要求低。10000 字大块完全可行。
润色是理解性重写，对注意力要求高，因此需要后文的小块（<=2000字）。

**无课件时的兜底手段**：
1. 纯程序化扫描转写稿全文，统计高频疑似乱码片段（如"斯普瑞布特"出现47次）
2. 将这些高频疑似词单独列给纠错 LLM："以下词汇在文中高频出现，请推断正确的技术术语"
3. LLM 自身知识兜底——在编程语境下，它能从上下文推断正确术语

纠错前：
> 我们先写一个斯普瑞布特的配置类，然后加上麦贝提斯普拉斯的分页拦截器...

纠错后：
> 我们先写一个 SpringBoot 的配置类，然后加上 MyBatisPlus 的分页拦截器...

### 0.3 课程全局摘要（采样 + 便宜LLM，一次性）

**目的**：让阶段1每块 LLM 知道这门课整体在讲什么。

**方案**：取纠错后文本的头尾各 ~3000 字 + 每隔 10000 字采样 500 字，
合并后约 6000-8000 字，让 DeepSeek-V4-Flash（non-thinking）生成一段课程概要。

输出格式：
```json
{
  "course_title": "Spring Boot 微服务实战",
  "overview": "本课程从零搭建一个Spring Boot微服务项目，涵盖自动配置原理、RESTful API设计、数据库集成、Docker部署...",
  "tech_stack": ["Spring Boot 3.x", "MyBatis-Plus", "MySQL", "Docker"],
  "chapters_summary": ["第1-3节讲项目搭建和自动配置", "第4-6节讲数据库集成..."]
}
```

### 0.4 话题边界检测（递归多层级切割）

**目的**：在语义边界上切分文本，每块控制在 1500-2000 字。

**方案**：多信号融合 + 递归切割。从高权重边界到低权重边界逐层筛选，
直到所有块都 <= 2000 字。

**切割信号（按权重降序）**：

| 优先级 | 信号类型 | 示例 |
|--------|---------|------|
| 最高 | 章节标记词 | "接下来我们讲..."、"下面来看..."、"最后一个部分" |
| 高 | 讲解->代码切换 | "我们写一下..."、"来看代码"、"运行一下" |
| 中 | 列举结构 | "首先...其次...最后..."、"第一步...第二步..." |
| 低 | 段落空行 | 连续两个换行，老师明显停顿 |

**递归流程**：
```
第1轮：检测高权重边界 -> 标记出 5-8 个大块（3000-8000字）
第2轮：对每个 > 2000 字的块，降低阈值继续检测 -> 1500-3000字
第3轮：对仍 > 2000 字的块，用段落空行强切 -> 1000-2000字
最终：N 个块，全部 <= 2000 字，全部在语义边界上
```

最坏情况：老师连讲 3000 字没有停顿——认栽用段落空行强切。
因阶段0.2已完成纠错，且阶段1有上下文，强切的质量损失可控。

### 0.5 RAG 索引构建（可选，一次性）

**目的**：为阶段1润色和阶段2代码注入提供检索能力。

**方案**：
- 代码文件按函数/类级切片，用 text-embedding-3-small 向量化
- 课件文件按段落/幻灯片切片，同样向量化
- 存入 ChromaDB
- 只建一次，代码/课件更新时增量重建

无代码/课件时跳过此步骤，阶段1和阶段2仅处理文本本身。

## 阶段1：全并行上下文润色

### 1.1 四层上下文设计

每块 LLM 调用时拿到四个层面的上下文，全部零额外 LLM 成本：

```
LLM 拿到当前块时，四个隐性问题各有一个答案：

问题1：这门课到底在讲什么？
  -> 全局摘要：阶段0.3生成，一次LLM调用，N次复用
  -> "Spring Boot 微服务实战，从自动配置、REST API到数据库集成、Docker部署"
  -> LLM 不会把"配置类"理解成普通的系统配置文件

问题2：前面讲到哪了？我从哪接话？
  -> 前块结尾80字："...所以 @ConditionalOnClass 就是条件装配的核心判断逻辑。"
  -> LLM 知道前面刚讲完什么，自然接续
  -> 这就是一个零成本的"前文 mini 摘要"

问题3：后面要讲什么？我这块要不要收尾？
  -> 后块开头60字："接下来我们动手写一个自定义的 Condition 实现类..."
  -> LLM 知道：后面要实操了，我这块不要做最终总结，留个自然过渡即可

问题4：我在全文结构中的位置？
  -> 位置标签："第 5 块 / 共 28 块"
  -> LLM 知道：我不是开头（不用做欢迎），不是结尾（不用做总结），我在中间
```

**为什么不需要全篇摘要链**：
- 远距离前文（10个块前讲过什么）与当前润色几乎无关——润色"MyBatis分页"时，
  不需要知道"Maven项目结构"的细节
- 全局摘要已覆盖"远处大概讲了什么"
- 前文结尾已覆盖“紧邻刚才讲到哪了”
- 没有信息盲区

````

## 你的位置
第 {chunk_index} 块 / 共 {total_chunks} 块

{retrieved_code_or_slides}

{current_chunk}

## 任务
将上面的文本润色为流畅的中文书面语：
- 去除口语化填充词（"那个"、"就是说"、"然后...然后..."）
- 整理为清晰的段落，但不要加任何 Markdown 标题
- 不要插入代码块
- 不要遗漏任何知识点和老师的讲解内容
- 不要做摘要压缩，只做语言精炼
- 确保与前文后文的过渡自然
```
````

### 1.3 RAG 结果指纹化（控制每块输入长度）

**问题**：RAG 检索到的完整代码文件可能达到 500-800 tokens，
加上四层上下文和当前原文，每块输入膨胀到 4000+ tokens，
LLM 要在"看代码"和"润色文字"之间分配注意力，润色质量下降。

**方案**：检索结果在喂给润色 LLM 之前做程序侧压缩——提取"代码指纹"：

```
完整代码（80行，800 tokens）:
  public class MybatisPlusConfig {
      @Bean
      public MybatisPlusInterceptor mybatisPlusInterceptor() {
          MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
          interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL));
          return interceptor;
      }
  }

压缩为指纹（~30 tokens）:
  代码: MybatisPlusConfig.java
  @Configuration @MapperScan
  mybatisPlusInterceptor()  // 分页插件配置
```

**压缩规则**：
- 代码：只保留类名、关键注解（`@Configuration`、`@Bean`）、方法签名，去掉方法体
- 课件：只保留标题和第一句话
- 相关性标注：在指纹后附一行说明这个文件与当前块的关联，
  如 "// 分页插件配置 — 老师在第5块正在讲解拦截器原理"

**效果**：参考资料从 500-800 tokens 压缩到 80-150 tokens，
每块总输入控制在 2400 tokens 以内，注意力集中在润色任务上。

### 1.4 并行策略

- 所有块同时调用 LLM（asyncio.gather），总耗时 = 最慢那块的时间
- 每个块的输入完全独立，不需要等前一个块完成
- 28 个块 -> 28 次并行调用，预计 30 秒内完成

### 1.5 关键约束

- 不加标题（结构是阶段2的事）
- 不插代码（代码注入是阶段2的事）
- 但必须阅读 RAG 代码指纹，以正确理解老师讨论的技术内容
- 润色目标是语言精炼，不是内容压缩

## 阶段2：全局结构化 + 代码注入

### 2.1 为什么代码注入在阶段2而不是阶段1

1. **叙事完整性**：老师写代码是渐进过程（先写一半->解释->补全->再解释），
   只有看到全文的 LLM 才能把握这个节奏
2. **避免重复插入**：同一段代码可能在多个块被讨论，分块插入会导致代码重复出现
   阶段2 的 LLM 能判断第一次插入完整代码，后续只需引用
3. **代码定位精度**：润色后的干净文本语义边界清晰，比原始转写稿更容易判断
   "老师正在讲这段代码"的精确位置

### 2.2 输入与任务

**输入**：
- 阶段1输出的完整干净文本（拼合所有块）
- 全部代码文件内容（RAG 索引中的完整代码）
- 课件中的关键内容

**模型**：DeepSeek-V4-Pro（thinking 模式，1M上下文窗口，推理链精准）

**任务**：
1. 给全文添加合理的层级标题（## / ### / ####，最多四级）
2. 在老师讲到具体代码的位置插入对应代码块（带语言标注）
3. 保留"先讲思路 -> 再写代码 -> 再解释"的叙事节奏
4. 如果老师是分步写代码的，笔记也要分步展示（先写一半，解释完再补全）

### 2.3 关键约束

- 不改变润色后的正文内容，只加标题和代码
- 代码块放在"老师正在讨论这段代码"的叙事位置，不是章节开头或结尾
- 不要插入正文中未提及的代码
- 标题层级合理，不过深
- 如果无代码/课件，此阶段仅做标题生成

**为什么阶段2要用 Pro + thinking？**：结构化需要同时把握微观（代码该插在哪句话后）和宏观（##/### 层级划分）。
Pro 的推理链能在做决策时自检"这个标题层级对吗？"，显著提升结构质量。

**为什么纠错和摘要不用 Pro？**：纠错和摘要都是轻量任务，Flash 完全胜任，不值得花 Pro 的钱。

------|------|------|
| 0.0 噪音清洗 | 简单 | 纯正则，一次性 |
| 0.1 术语词典 | 简单 | 纯正则提取，一次性 |
| 0.2 全文纠错 | 简单 | 机械任务，Prompt 简单 |
| 0.3 全局摘要 | 简单 | 一次便宜 LLM 调用 |
| 0.4 递归边界检测 | 中等 | 多层级规则调优 |
| 0.5 RAG 索引 | 中等 | 多格式解析 + 合理切片粒度 |
| 阶段1 并行润色 | 中等 | Prompt 迭代量最大（不遗漏 vs 不过度） |
| 阶段2 标题添加 | 简单 | 依赖大窗口模型能力 |
| 阶段2 代码注入 | 中等 | 代码定位精度是核心指标 |
